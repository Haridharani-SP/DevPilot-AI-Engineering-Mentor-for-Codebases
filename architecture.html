System architecture

The architecture is designed as a full knowledge-to-intelligence pipeline: collect engineering context, convert it into searchable knowledge, generate grounded answers, and transform repeated developer questions into onboarding insight.

How DevPilot works end-to-end

Architecture Diagram

This diagram shows how DevPilot moves from raw engineering artifacts to intelligent onboarding support and team-level learning analytics.

Layer 01

Knowledge sources

DevPilot begins with real engineering knowledge: project documentation, GitHub repositories, and codebase files. These are the raw sources the system depends on to understand how the software actually works.

Layer 02

Processing layer

The processing pipeline performs code ingestion, chunking, and embedding generation. This converts large, unstructured engineering content into smaller semantic units that can be searched intelligently.

Layer 03

Knowledge storage

DevPilot stores semantic knowledge in a FAISS vector database and stores structured metadata in SQLite. Together, these layers support both retrieval accuracy and traceability back to real files and sources.

Layer 04

Intelligence layer

The RAG retrieval pipeline pulls the most relevant engineering context, the language model generates a grounded answer, and the knowledge gap analyzer identifies recurring areas of confusion from user interactions.

Layer 05

Learning analytics

Developer questions are not treated as isolated events. DevPilot converts them into insights about confusing modules, missing documentation, and individual learning patterns, giving the team real visibility into onboarding friction.

Layer 06

Application layer

FastAPI exposes the core intelligence as services, while the React dashboard and Slack integration bring that intelligence into developer workflows. The result is a usable system, not just a backend model.
